# 服务接口契约

## 1. 约定

本文档定义 Spring Boot Platform 与 LangGraph Agent Engine 之间的接口契约。

原则：

- 所有接口使用 JSON。
- 所有请求必须携带 `runId`。
- 内部接口必须携带 `X-Internal-Token`。
- 错误响应必须结构化。
- 时间使用 ISO-8601 或毫秒时间戳，避免模糊格式。

通用错误格式：

```json
{
  "success": false,
  "errorCode": "SQL_COMPILE_ERROR",
  "message": "Unknown column 'xxx'",
  "details": {}
}
```

## 2. Spring Boot 对外接口

### 2.1 发起分析

```http
POST /api/analyze
Content-Type: application/json
```

请求：

```json
{
  "userId": "demo",
  "message": "为什么国庆后美食视频完播率下降了",
  "bypassCache": false
}
```

响应：

```json
{
  "runId": "run_20260611_001",
  "status": "SUCCESS",
  "report": {
    "summary": "...",
    "period": "2023-10-01~2023-10-31",
    "metrics": [],
    "charts": [],
    "recommendations": []
  }
}
```

### 2.2 查询运行列表

## 3. Spring Boot 内部接口

### 3.1 创建 Run

```http
POST /internal/runs
X-Internal-Token: ${INTERNAL_API_TOKEN}
```

请求：

```json
{
  "runId": "run_20260611_001",
  "userId": "demo",
  "question": "为什么国庆后美食视频完播率下降了"
}
```

响应：

```json
{
  "success": true,
  "runId": "run_20260611_001"
}
```

### 3.2 开始节点

```http
POST /internal/runs/{runId}/nodes
```

请求：

```json
{
  "nodeName": "SQL_GENERATE",
  "inputPayload": {
    "question": "...",
    "schemaContext": "..."
  }
}
```

响应：

```json
{
  "success": true,
  "nodeId": 1001
}
```

### 3.3 完成节点

```http
PATCH /internal/runs/{runId}/nodes/{nodeId}
```

请求：

```json
{
  "status": "SUCCESS",
  "outputPayload": {
    "sql": "SELECT ..."
  },
  "modelName": "deepseek-chat",
  "promptTokens": 1200,
  "completionTokens": 300
}
```

响应：

```json
{
  "success": true
}
```

失败节点：

```json
{
  "status": "FAILED",
  "errorMessage": "SQL compile error"
}
```

跳过节点：

```json
{
  "status": "SKIPPED",
  "errorMessage": "RAG confidence < 0.3"
}
```

### 3.4 SQL 执行

```http
POST /internal/sql/execute
```

请求：

```json
{
  "runId": "run_20260611_001",
  "userId": "demo",
  "question": "为什么国庆后美食视频完播率下降了",
  "sql": "SELECT date, category, total_plays FROM metric_daily WHERE category = '美食'",
  "purpose": "查询美食分类每日播放趋势",
  "allowHighRisk": false
}
```

成功响应：

```json
{
  "success": true,
  "sql": "SELECT ...",
  "columns": ["date", "category", "total_plays"],
  "rows": [
    {"date": "2023-10-01", "category": "美食", "total_plays": 100}
  ],
  "rowCount": 31,
  "truncated": false,
  "warnings": [],
  "riskLevel": "LOW",
  "accessedTables": ["metric_daily"],
  "durationMs": 23
}
```

SQL 编译失败：

```json
{
  "success": false,
  "sql": "SELECT bad_col FROM metric_daily",
  "errorCode": "SQL_COMPILE_ERROR",
  "error": "Unknown column 'bad_col'",
  "warnings": [],
  "riskLevel": "LOW",
  "accessedTables": ["metric_daily"],
  "durationMs": 5
}
```

高风险 SQL：

```json
{
  "success": false,
  "sql": "SELECT * FROM user_behavior_fact",
  "errorCode": "SQL_HIGH_RISK",
  "error": "Large table scan requires approval",
  "warnings": [
    "Query scans user_behavior_fact without time range",
    "Query has no LIMIT"
  ],
  "riskLevel": "HIGH",
  "accessedTables": ["user_behavior_fact"]
}
```

## 5. 版本兼容

接口版本通过路径控制：

```text
/api/v1/analyze
/internal/v1/sql/execute
```

初期可以省略 `/v1`，但正式整理前必须加版本号。

## 6. 超时建议

| 接口 | 超时 |
|---|---:|
| `/api/analyze` 同步 | 60s |
| `/internal/sql/execute` | 20s |
| `/internal/schema/relevant` | 10s |
| `/internal/metrics/{name}` | 5s |
| LangGraph LLM 节点 | 60s |
| RAG 节点 | 30s |

同步分析后期建议改成异步：

```text
POST /api/analyze -> 返回 runId
GET /api/runs/{runId} -> 查询状态
SSE /api/runs/{runId}/events -> 推送进度
```

## 7. 安全要求

- 内部接口必须鉴权。
- SQL 请求必须记录 userId 和 runId。
- 不允许 LangGraph 直接拿数据库账号。
- 不允许在 payload 中传 API Key。
- 审计日志不可被普通用户删除。

## 8. ChatBI 主链路新增接口

本节补充 ChatBI 主链路需要新增或强化的内部接口。完整设计见 `docs/ChatBI主链路开发设计.md`。

### 8.1 SQL 硬校验

```http
POST /internal/sql/validate
Content-Type: application/json
X-Internal-Token: ${INTERNAL_API_TOKEN}
```

Request:

```json
{
  "runId": "run_xxx",
  "userId": "demo",
  "question": "分析各分类播放量趋势",
  "sql": "SELECT date, category, total_plays FROM metric_daily",
  "purpose": "查询各分类每日播放量趋势"
}
```

Response:

```json
{
  "pass": true,
  "riskLevel": "LOW",
  "errorCode": null,
  "reason": null,
  "suggestion": null,
  "violations": [],
  "accessedTables": ["metric_daily"],
  "accessedColumns": ["date", "category", "total_plays"]
}
```

失败响应:

```json
{
  "pass": false,
  "riskLevel": "MEDIUM",
  "errorCode": "MISSING_TIME_RANGE",
  "reason": "明细表查询缺少时间范围",
  "suggestion": "补充时间范围，或改用 metric_daily 聚合表",
  "violations": [
    {
      "ruleCode": "DETAIL_TABLE_REQUIRES_TIME_RANGE",
      "severity": "ERROR",
      "message": "play_detail requires time range"
    }
  ],
  "accessedTables": ["play_detail"],
  "accessedColumns": ["content_id", "play_duration"]
}
```

### 8.2 SQL 结果 DQ 审核

```http
POST /internal/dq/sql-result/check
Content-Type: application/json
X-Internal-Token: ${INTERNAL_API_TOKEN}
```

Request:

```json
{
  "runId": "run_xxx",
  "userId": "demo",
  "question": "分析各分类播放量趋势",
  "sql": "SELECT category, total_plays FROM metric_daily",
  "schemaContext": "metric_daily: date, category, total_plays",
  "attemptNo": 1,
  "queryResult": {
    "columns": ["category", "total_plays"],
    "rows": [
      {
        "category": "美食",
        "total_plays": 1000
      }
    ],
    "rowCount": 1
  }
}
```

Response:

```json
{
  "pass": false,
  "severity": "HIGH",
  "reason": "用户问题要求分析趋势，但查询结果缺少 date 字段",
  "suggestion": "按 date 和 category 分组返回 total_plays，并按 date 升序排序",
  "missingDimensions": ["date"],
  "missingMetrics": [],
  "anomalySignals": ["MISSING_TIME_DIMENSION"],
  "confidence": 0.9
}
```
