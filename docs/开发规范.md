# 开发规范

## 1. 总原则

本项目目标是企业级 DataAgent，不是一次性 Demo。所有改动应遵守：

1. **先结构化，再智能化**
   关键状态必须用 DTO/TypedDict 表达，不能只靠自然语言字符串。

2. **平台能力和 Agent 推理解耦**
   SQL 执行、权限、审计、缓存、指标字典属于 Spring Boot 平台层；Agent 只做决策和编排。

3. **所有节点可追踪**
   每个 Agent 节点必须记录状态、输入、输出、耗时和错误。

4. **所有安全控制确定性优先**
   SQL 安全、权限、脱敏不能交给 LLM 判断。

5. **每个阶段都保持可运行**
   不允许长时间处于半迁移、主链路不可启动的状态。

## 2. 分支与提交规范

推荐分支：

```text
codex/platform-run-trace
codex/sql-gateway
codex/langgraph-engine
codex/eval-system
codex/dashboard
```

### 2.1 提交粒度

提交必须以**一个最小可验证功能**为边界。宁可多 commit，也不要把多个能力塞进一个大提交。

推荐拆分方式：

```text
1 个表结构变更 = 1 个 commit
1 个 DTO/枚举补充 = 1 个 commit
1 个 Service 能力 = 1 个 commit
1 个 Controller 接口 = 1 个 commit
1 个 Agent 节点迁移 = 1 个 commit
1 组对应测试 = 1 个 commit
1 次文档同步 = 1 个 commit
```

提交粒度规则：

- 一个提交只完成一个明确能力。
- 每个提交应尽量能独立编译。
- 每完成一个最小功能点就提交，不等到整个阶段结束。
- 不把格式化、重命名、功能修改混在一起。
- 不把文档、测试、业务逻辑的大量无关改动混在一起。
- 不提交真实密钥、数据库密码、本地 IDE 配置。

好例子：

```text
feat(run): add agent_run and agent_run_node schema
feat(run): add run status enums and DTOs
feat(run): implement AgentRunTraceService
feat(run): expose admin run query endpoints
test(run): add trace service tests
docs(run): document run trace lifecycle
```

坏例子：

```text
feat: finish stage one
```

这个提交名太大，无法看出具体改了什么，也不利于回滚。

### 2.2 提交信息格式

提交信息格式：

```text
feat(platform): add agent run trace tables
feat(sql): introduce structured sql execution result
fix(config): remove default api key from application config
test(agent): add sql retry graph test
docs(architecture): add langgraph migration design
```

推荐类型：

```text
feat      新功能
fix       缺陷修复
refactor  内部重构，不改变行为
test      测试
docs      文档
chore     构建、脚本、依赖、配置整理
perf      性能优化
security  安全相关修复
```

### 2.3 提交前检查

提交前至少确认：

```text
git diff --check
mvn test
```

如果当前提交只改文档，可以不跑完整测试，但提交信息里应清楚标记 `docs(...)`。

如果测试因为本地外部依赖失败，必须在提交说明或 PR 描述里写清楚：

```text
验证命令：
- mvn test

结果：
- 失败，原因是本地 MySQL 未启动
```

### 2.4 回滚友好原则

每个 commit 都应能被单独 revert。为了做到这一点：

- 不在一个 commit 里同时改多个阶段的功能。
- 不把临时调试代码和正式功能混在一起。
- 不提交大规模无关格式化。
- 数据库表结构变更和依赖该表的业务代码可以分两个 commit，但顺序必须保证主分支始终可编译。

### 2.5 阶段性开发建议

以阶段一 Run Trace 为例，推荐拆成：

```text
feat(run): add run trace schema
feat(run): add run status enums and DTOs
feat(run): implement trace write service
feat(run): implement run query service
feat(run): expose admin run endpoints
feat(run): attach runId to analysis report
feat(run): record analyze lifecycle
test(run): verify application context with run trace
docs(run): update platform design notes
```

这比一个 `feat: add run trace` 更容易 review、定位问题和回滚。

## 3. 配置规范

### 3.1 禁止项

禁止在仓库中提交：

- 真实 API Key。
- 真实数据库密码。
- 生产 Redis 地址。
- 私有模型服务 token。
- 本地绝对路径。

### 3.2 Profile

必须至少保留：

```text
application.yml
application-dev.yml
application-test.yml
application-prod.yml
```

要求：

- `application.yml` 只写无默认值的环境变量占位。
- `application-dev.yml` 可以有本地开发默认值，但不能有真实密钥。
- `application-test.yml` 必须避免真实外部依赖。
- `application-prod.yml` 默认关闭 prompt/completion 内容日志。

### 3.3 环境变量

统一命名：

```text
AI_API_KEY
AI_BASE_URL
AI_STRONG_MODEL
AI_CHEAP_MODEL
DB_URL
DB_USERNAME
DB_PASSWORD
REDIS_HOST
REDIS_PORT
OLLAMA_BASE_URL
OLLAMA_EMBED_MODEL
INTERNAL_API_TOKEN
LANGGRAPH_BASE_URL
```

## 4. Java 开发规范

### 4.1 包结构

新增平台能力优先放到明确模块：

```text
controller
service
domain
dto
config
security
```

不要继续把所有逻辑塞进 Agent 类。

### 4.2 DTO 规范

跨层返回值必须使用 DTO，不返回裸 `Map<String,Object>`，除非是通用 payload。

推荐：

```java
public record SqlExecuteResult(
        boolean success,
        String sql,
        List<String> columns,
        List<Map<String, Object>> rows,
        int rowCount,
        boolean truncated,
        List<String> warnings,
        String error,
        String riskLevel,
        long durationMs
) {}
```

### 4.3 Service 规范

Service 应该单一职责。

不推荐：

```text
SqlExecutionTool 同时做 Parser、规则、EXPLAIN、执行、缓存、审计、熔断。
```

推荐：

```text
SqlExecutionService
SqlParserValidator
SqlRulesChecker
SqlPlanAnalyzer
SqlAuditService
SqlResultCache
```

### 4.4 并发规范

禁止使用全局 static 保存请求上下文。

禁止：

```java
private static String lastExecutedSql;
```

推荐：

```text
runId + nodeId + structured result
```

所有异步任务必须使用受控线程池，不直接使用 common pool。

### 4.5 异常规范

平台异常使用结构化错误码。

示例：

```text
SQL_SYNTAX_ERROR
SQL_COMPILE_ERROR
SQL_HIGH_RISK
PERMISSION_DENIED
METRIC_NOT_FOUND
SCHEMA_LOAD_FAILED
AGENT_NODE_FAILED
```

LLM 可读错误和系统错误要分开：

- 系统日志记录详细异常。
- 返回给 Agent 的错误必须简洁、可修正。

## 5. Python LangGraph 开发规范

### 5.1 State 规范

所有节点必须读写 `DataAgentState`。

禁止：

- 使用模块级全局变量保存状态。
- 节点之间隐式共享对象。
- 在 state 中存 API Key。

### 5.2 节点规范

每个节点一个明确职责：

```text
router_node: 只路由
schema_node: 只拿 schema
sql_generate_node: 只生成 SQL
sql_execute_node: 只调用平台执行 SQL
sql_validate_node: 只判断是否重试/继续
```

节点函数必须满足：

- 输入 state。
- 输出更新后的 state。
- 不直接写数据库，除非通过 PlatformClient。
- 异常向上抛，由 wrapper 记录失败。

### 5.3 Prompt 规范

Prompt 放在 `app/prompts/`。

每个 Prompt 必须包含：

- 角色。
- 输入变量。
- 输出格式。
- 禁止行为。
- 失败时输出约定。

输出必须尽量 JSON 化。

### 5.4 PlatformClient 规范

LangGraph 访问 Spring Boot 只能通过 `PlatformClient`。

禁止在任意 Agent 文件里直接写 HTTP 调用。

推荐：

```python
await platform.execute_sql(request)
await platform.start_node(run_id, node_name, input_payload)
await platform.finish_node(node_id, output_payload)
```

## 6. SQL 安全规范

所有 SQL 必须经过 Spring Boot SQL Gateway。

必须检查：

- 是否 SELECT。
- 是否多语句。
- 是否访问未授权表。
- 是否访问敏感字段。
- 是否缺少 LIMIT。
- 是否缺少时间范围。
- 是否 EXPLAIN 扫描过大。
- 是否聚合口径不明确。

高风险 SQL 不得自动执行，必须进入审批。

## 7. Run Trace 规范

每个节点必须记录：

```text
run_id
node_name
status
input_payload
output_payload
error_message
duration_ms
model_name
prompt_tokens
completion_tokens
retry_count
```

payload 规范：

- 默认保存结构化 JSON。
- 超过限制时保存 preview。
- 不保存密钥。
- 用户隐私字段需要脱敏。

## 8. 测试规范

### 8.1 Java 测试

必须覆盖：

- SQL 只读校验。
- SQL 规则校验。
- SQL 高风险判断。
- AgentRunTraceService。
- MetricCatalogService。
- SchemaCatalogService。

命令：

```bash
mvn test
```

### 8.2 Python 测试

必须覆盖：

- Graph 路由。
- SQL retry。
- RAG confidence 分支。
- DBQA 补充查询。
- Human approval 暂停与恢复。
- PlatformClient 错误处理。

命令：

```bash
pytest
```

### 8.3 评测

评测不等于单元测试。

评测用于回答：

- SQL 准确率是多少？
- RAG 命中率是多少？
- 报告质量如何？
- 平均耗时多少？
- Token 成本多少？

评测报告必须输出到：

```text
docs/eval-report.md
```

## 9. 日志规范

日志必须包含：

```text
run_id
node_name
user_id
duration_ms
status
```

禁止默认记录完整 Prompt 和 Completion 到生产日志。

生产环境：

```yaml
spring:
  ai:
    chat:
      client:
        observations:
          log-prompt: false
          log-completion: false
```

## 10. 文档规范

新增核心能力时必须更新：

- 架构文档。
- 接口契约。
- 配置说明。
- 评测说明。

文档应说明：

- 为什么做。
- 怎么设计。
- 怎么使用。
- 怎么验证。
- 已知限制。

## 11. Definition of Done

一个功能完成必须满足：

- 代码已实现。
- 单元测试或集成测试已覆盖。
- 本地验证通过。
- 文档已更新。
- 没有真实密钥。
- 没有破坏主链路。
- 失败场景有明确处理。

对于 Agent 节点，还必须满足：

- 节点状态写入 Run Trace。
- 输入输出结构化。
- 可独立测试。
- 有 fallback 或明确失败策略。
